บทเรียนที่ 1: ภาพรวมสถาปัตยกรรม IoT Hub

ยินดีต้อนรับสู่บทเรียนแรก! ในบทเรียนนี้เราจะทำความเข้าใจเกี่ยวกับหลักการของ IoT Hub ข้อเปรียบเทียบฮาร์ดแวร์ และโครงสร้างการจัดเก็บข้อมูล

Part 1: IoT Hub Concept & System Diagram

หัวใจสำคัญของระบบ IoT Hub คือการผสานเทคโนโลยีระหว่างเซ็นเซอร์ปลายทาง โบรกเกอร์กระจายข่าวสาร เว็บเซิร์ฟเวอร์ ฐานข้อมูล และส่วนการแสดงผลแดชบอร์ดผ่านหน้าเว็บ โดยระบบทำงานประสานกันตามขั้นตอนดังนี้:

Microcontrollers
(ESP32 / ESP8266)
➔ MQTT
MQTT Broker
(Mosquitto on Pi)
➔ Sub
FastAPI Backend
(Python on Pi)
➔ SQL
SQLite Database
(Local DB on Pi)
⬌ HTTP
Web Dashboard
(Browser UI)

รายละเอียดการประมวลผลทั้ง 5 ส่วน:

  • Microcontrollers (ESP32/ESP8266): ทำหน้าที่ดึงข้อมูลจากฮาร์ดแวร์เซ็นเซอร์ภายนอก จากนั้นแพ็กเป็น JSON string และพับลิช (Publish) ไปยังโบรกเกอร์ผ่าน Wi-Fi
  • MQTT Broker (Mosquitto): เป็นตัวกลางจัดการคิวข้อความแบบเบาบาง (Lightweight Pub/Sub) ไม่เก็บข้อมูลลงดิสก์ถาวร แต่เน้นความไวในการกระจายข้อมูล
  • FastAPI (Backend Service): ทำหน้าที่เป็นไคลเอนต์จำลองรอรับข้อมูล (Subscribe) รันตลอดเวลาเพื่อดักรับคีย์ข้อมูลเซ็นเซอร์แล้วจัดเก็บลงฐานข้อมูล พร้อมทั้งให้บริการ API endpoints สำหรับดึงข้อมูลและส่งคำสั่งควบคุม
  • SQLite Database: จัดเก็บข้อมูลทั้งหมดลงในไฟล์แบบถาวร เพื่อให้เซิร์ฟเวอร์สามารถเรียกข้อมูลสถิติย้อนหลังได้
  • Web Dashboard (Browser UI): หน้าเว็บแอปพลิเคชันฝั่งผู้ใช้งาน (Client) ที่เชื่อมต่อเข้ากับ FastAPI Backend ผ่านโปรโตคอล HTTP เพื่อเรียกขอข้อมูลแบบเป็นระยะ (Polling) มาแสดงผลเป็นกราฟิกสวยงาม และส่งคำสั่งควบคุมย้อนกลับไปยังบอร์ดไมโครคอนโทรลเลอร์

Part 2: Platform Selection (Ubuntu Server vs Raspberry Pi OS)

การเลือกใช้ระบบปฏิบัติการและบอร์ดคอมพิวเตอร์ขนาดเล็กในการทำ IoT Hub:

  • Raspberry Pi OS (Debian-based): ออกแบบมาเพื่อฮาร์ดแวร์ Raspberry Pi โดยเฉพาะ มีไดรเวอร์จัดการการใช้พลังงานและการคุมความร้อนที่ดีมาก เหมาะอย่างยิ่งสำหรับ Raspberry Pi Zero 2 W ที่มี RAM เพียง 512MB
  • Ubuntu Server (Ubuntu-based): เหมาะสำหรับกรณีรันบนคอมพิวเตอร์ทั่วไป (x86) หรือ Mini PC ที่เปิดทิ้งไว้ มีแพ็คเกจซอฟต์แวร์เวอร์ชันค่อนข้างใหม่กว่า แต่ต้องการแรมในการเปิดระบบ (Base system) มากขึ้น
💡 คำแนะนำระดับโปรดักชัน: หากระบบมีทรัพยากรน้อยกว่า 1GB RAM การติดตั้ง OS แบบปราศจาก GUI (Headless Server) จะช่วยคืนทรัพยากร CPU และ RAM ให้มาใช้งานฐานข้อมูลและเซิร์ฟเวอร์ได้อย่างเต็มที่

Part 3: Data Memory Layout (Scattered Packet vs Contiguous DB Pages)

เพื่อให้เข้าใจความแตกต่างของโมเดลข้อมูล เราแยกโมเดลการประมวลผลออกเป็น 2 ส่วนตามลักษณธการกระจายในหน่วยความจำ:

  • Scattered Packet (ฝั่งเครือข่าย/ไคลเอนต์): ข้อมูลที่วิ่งอยู่บน MQTT หรือ RAM ในบอร์ด ESP32 จะเป็นข้อมูลกระจัดกระจาย (Scattered) มีขนาดเล็ก วิ่งเป็นคีย์-ค่าสั้นๆ ผ่านเธรดการประมวลผลที่เน้นส่งเร็วและทำลายทิ้งทันที
  • Contiguous DB Pages (ฝั่งฐานข้อมูล/ดิสก์): เมื่อข้อมูลมาถึง SQLite จะถูกจับมารวมและเรียงตัวต่อเนื่องลงบนหน้ากระดาษจำลอง (Pages) ขนาด 4KB ใน Harddrive/SD Card การเรียงตัวแบบเป็นระเบียบต่อเนื่อง (Contiguous) ช่วยให้สามารถสแกนสถิติช่วงเวลาย้อนหลัง (Query) ทำได้อย่างรวดเร็วมาก

ตรวจสอบความเข้าใจ (Quick Quiz)

เพราะเหตุใดข้อมูลในรูปแบบ Scattered (เช่น Payload MQTT ที่ส่งมาจาก ESP32) จึงไม่ควรนำมาเขียนลงในดิสก์ตรงๆ ทุกครั้งที่ส่งเข้ามาโดยไม่มีการตั้งค่าป้องกัน?

A) เพราะเครือข่ายจะช้าลงอย่างมาก
B) เพราะการเขียนไฟล์ขนาดเล็กและไม่ต่อเนื่องถี่เกินไป จะทำให้ SD Card สึกหรอและพังเร็วขึ้น
