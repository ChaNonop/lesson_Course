บทเรียนที่ 3: พื้นฐาน Docker และการใช้งานในระบบ IoT Hub

ในบทเรียนนี้ เราจะมาทำความเข้าใจว่า Docker คืออะไร แตกต่างจาก Virtual Machine อย่างไร เจาะลึกแนวคิดสำคัญ เช่น Image, Container, Volume, Network และเรียนรู้คำสั่งควบคุมพื้นฐานเพื่อพร้อมรันบริการต่าง ๆ บนบอร์ด IoT Hub

Part 1: Docker คืออะไร? (Containers vs Virtual Machines)

Docker เป็นแพลตฟอร์มในการบรรจุแอปพลิเคชันและสภาพแวดล้อมที่จำเป็นทั้งหมด (Libraries, Python runtime, Config) เข้าไว้ด้วยกันเป็นยูนิตเดี่ยวที่เรียกว่า Container เพื่อให้แอปพลิเคชันทำงานได้เหมือนกันในทุกระบบปฏิบัติการ ไม่ว่าจะเป็นบนโน้ตบุ๊กทีมพัฒนา หรือบนบอร์ดคอมพิวเตอร์ขนาดเล็ก (Raspberry Pi/Mini PC)

เปรียบเทียบกับ Virtual Machine (VM)

  • Virtual Machine (เช่น VMware, VirtualBox): จำลองฮาร์ดแวร์ขึ้นมาใหม่ทั้งหมด และต้องลงระบบปฏิบัติการแยกเฉพาะ (Guest OS) ทับลงไป ทำให้กินทรัพยากรสูง (RAM หลาย GB) และบูตช้าเพราะต้องโหลด OS ใหม่
  • Docker Container: ใช้ Kernel ของระบบปฏิบัติการหลัก (Host OS) ร่วมกัน และทำลายสิ่งกีดขวางเฉพาะพื้นที่ทำงานของแอปพลิเคชัน (OS virtualization) ทำให้มีขนาดเล็กมาก (หลักสิบ/ร้อย MB) บูตเสร็จภายในเศษเสี้ยววินาที และใช้แรมต่ำมาก เหมาะสมอย่างยิ่งสำหรับบอร์ด IoT ที่มีแรมจำกัด
💡 มุมมองประสิทธิภาพบน IoT: การใช้ Docker ช่วยประหยัด RAM บนบอร์ด Raspberry Pi ไปได้มหาศาลเมื่อเทียบกับ VM ทำให้เราสามารถรันทั้ง MQTT Broker และ FastAPI Backend ได้พร้อมกันในเนื้อที่แรมไม่กี่ร้อยเมกะไบต์

Part 2: 4 เสาหลักของ Docker (Core Concepts)

การใช้งาน Docker จำเป็นต้องเข้าใจ 4 ออบเจกต์หลักที่ทำงานสัมพันธ์กันดังนี้:

  1. Docker Image (พิมพ์เขียว): ไฟล์แม่แบบที่เก็บรหัสซอร์สโค้ด รันไทม์ ไลบรารี และคอนฟิกของแอปพลิเคชันไว้ มีสถานะ Read-only (อ่านอย่างเดียว) เปรียบเสมือนแผ่น CD-ROM ติดตั้งโปรแกรม
  2. Docker Container (การสวมหัวรัน): ตัวประมวลผลที่เกิดจากการนำพิมพ์เขียว (Image) มารันให้ทำงานจริง เป็นพื้นที่เขียนอ่านได้ชั่วคราว (Writable Layer) โดยตู้คอนเทนเนอร์นี้จะถูกตัดขาดจากภายนอกเพื่อความปลอดภัย
  3. Docker Volume (ระบบยึดข้อมูลถาวร): เนื่องจากธรรมชาติของ Container จะสูญเสียข้อมูลทั้งหมดข้างในทันทีเมื่อ Container ถูกลบหรือรีสตาร์ท (Stateless) เราจึงต้องเมาท์โฟลเดอร์ภายนอก (Host) เข้าไปในตู้คอนเทนเนอร์เพื่อให้ข้อมูลต่างๆ เช่น ไฟล์ฐานข้อมูล SQLite หรือไฟล์ Log ถูกเก็บไว้ได้อย่างถาวรแม้ตู้จะระเบิดไป
  4. Docker Network (ระบบเครือข่ายส่วนตัว): ช่วยเชื่อมสายสัญญาณจำลองระหว่าง Container ต่างๆ ให้อยู่ในวงเน็ตเวิร์กเดียวกัน (Bridge Network) เพื่อให้สามารถเชื่อมต่อข้ามกล่องกันได้ เช่น FastAPI Backend คุยกับ Mosquitto Broker โดยอ้างอิงชื่อบริการ แทนการอ้างอิง IP Address

Part 3: คำสั่งควบคุม Docker ที่ใช้งานบ่อย (Essential Commands)

คำสั่ง CLI พื้นฐานที่นักพัฒนา IoT Hub ต้องฝึกฝนและหัดสั่งงานผ่าน Terminal:

1. ดึงภาพและเปิดตู้ทำงาน (docker run)

docker run -d --name my_mosquitto -p 1883:1883 -v ./config:/mosquitto/config eclipse-mosquitto

คำอธิบายพารามิเตอร์:

  • -d: รันแบบเบื้องหลัง (Background/Detached)
  • --name: ตั้งชื่อตู้คอนเทนเนอร์ให้จำง่าย
  • -p 1883:1883: จับคู่พอร์ต (Port mapping) จากพอร์ตภายนอกของเครื่อง Host ไปยังพอร์ตภายในของตู้
  • -v: เชื่อมโยงโฟลเดอร์เพื่อจัดเก็บข้อมูลถาวร (Volume Mount)

2. ตรวจสอบสถานะการทำงาน (docker ps)

ใช้ส่องดูว่าตู้คอนเทนเนอร์ใดเปิดทำงานอยู่บ้าง หรือมีตู้ใดที่กำลังปิดการทำงานกระทันหัน:

# ตรวจสอบตู้ที่กำลังรันอยู่ปัจจุบัน
docker ps

# ตรวจสอบตู้ทั้งหมด รวมถึงตู้ที่หยุดทำงานไปแล้ว
docker ps -a

3. ตรวจสอบบันทึกเหตุการณ์การประมวลผล (docker logs)

ใช้ดีบักข้อผิดพลาด เช่น ดูว่าทำไมบอร์ดต่อเข้า broker ไม่ติด หรือเซิร์ฟเวอร์ไพทอนขึ้น error ตัวใด:

docker logs -f my_mosquitto

(กด Ctrl+C เพื่อออกจากหน้าแสดง Log)

4. การควบคุมและทำลาย (stop / start / rm)

# หยุดการทำงานชั่วคราว
docker stop my_mosquitto

# เริ่มทำงานใหม่อีกครั้ง
docker start my_mosquitto

# ลบตู้คอนเทนเนอร์ทิ้ง (ต้องหยุดตู้ก่อนลบ)
docker rm my_mosquitto

Part 4: Docker Compose เบื้องต้น (Multi-Container Orchestration)

เมื่อแอปพลิเคชันมีโครงสร้างซับซ้อนและประกอบด้วยหลายส่วนทำงานร่วมกัน (เช่น Mosquitto + FastAPI + SQLite) การพิมพ์คำสั่ง docker run ทีละตู้เป็นเรื่องยุ่งยากและเสียเวลา เราจึงใช้ Docker Compose ในการเขียนคอนฟิกแบบประกาศตัวแปรตัวเดียวลงในไฟล์ docker-compose.yml:

version: '3.8'
services:
  mqtt:
    image: eclipse-mosquitto:latest
    ports:
      - "1883:1883"
  backend:
    image: fastapi_app:latest
    ports:
      - "8000:8000"
    depends_on:
      - mqtt

ด้วยโครงสร้างนี้ เราสามารถสั่งรันบริการทั้งหมดขึ้นมาพร้อมเชื่อมต่อเครือข่ายให้เสร็จสรรพผ่านคำสั่งเดียว:

# สั่งรันบริการทั้งหมดขึ้นมาในโหมดเบื้องหลัง
docker compose up -d

# สั่งปิดและทำลายตู้ในโปรเจคทั้งหมดทิ้งพร้อมกัน
docker compose down

🧠 ตรวจสอบความเข้าใจ (Quick Quiz)

ทำไมเราจึงจำเป็นต้องใช้งาน Docker Volume (ตัวแปร -v) เมื่อต้องการทำระบบ MQTT Broker หรือ SQLite Database ใน Docker Container?

A) เพื่อช่วยเปิดพอร์ตเครือข่ายให้ ESP32 สามารถติดต่อเข้ามาที่เซิร์ฟเวอร์ได้
B) เพื่อรักษาฐานข้อมูลหรือไฟล์คอนฟิกให้ยังคงอยู่อย่างถาวร แม้ตู้คอนเทนเนอร์จะถูกหยุดทำงาน หรือลบทิ้งไป
